## How It Works

### Normal Operation (PRIMARY Healthy)
1. PRIMARY actively trades on all symbols
2. PRIMARY syncs state to BACKUP every 5 seconds
3. BACKUP monitors PRIMARY's health every 10 seconds
4. BACKUP sees PRIMARY as healthy → stays in monitor mode
5. BACKUP has current copy of account state (cash, positions)

### Failover (PRIMARY Fails)
1. BACKUP's health check fails for 10 seconds
2. BACKUP detects PRIMARY failure
3. BACKUP auto-enables autonomous trader
4. BACKUP starts placing trades immediately
5. State sync continues (now one-directional until PRIMARY recovers)

### Recovery (PRIMARY Comes Back)
1. BACKUP's health check succeeds
2. BACKUP detects PRIMARY recovery
3. BACKUP auto-disables autonomous trader
4. BACKUP syncs to PRIMARY's latest state
5. PRIMARY resumes trading

## Known Limitations (Phase 1)

⚠️ **Trade History Separate**
- Each machine has independent SQLite database
- Trades executed on PRIMARY before failure not visible on BACKUP
- Impact: LOW for Phase 1 (failover is typically <10 minutes)
- Solution: PostgreSQL in Phase 3

⚠️ **Config Changes Manual**
- Config changes must be deployed to both simultaneously
- No auto-sync for config updates
- Impact: LOW (config changes are rare)
- Solution: Config sync endpoint in Phase 2

⚠️ **SSH Tunnel Manual Startup**
- Tunnel runs in background but must be restarted if system reboots
- Impact: LOW (system reboots are rare)
- Solution: Systemd service in Phase 2
